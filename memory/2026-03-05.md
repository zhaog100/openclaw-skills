## 13:25 - 上下文监控脚本修复 ⭐⭐⭐⭐⭐

### 问题诊断
- **症状**：飞书对话出现 `model_context_window_exceeded` 错误
- **表象**：定时任务在运行，但无法防止超限
- **根本原因**：
  1. 监控脚本只"数文件"，没有真正检查上下文
  2. 10分钟间隔太长，会话在两次检查之间爆了
  3. 外部监控是"被动检查"，不是"主动预防"

### 解决方案（已完成）
**改进监控脚本**：
- ✅ 从"数文件"改为"调用OpenClaw API"
- ✅ 使用 `openclaw sessions --active 120 --json` 获取真实会话信息
- ✅ 提取 `totalTokens` 和 `contextTokens` 计算使用率
- ✅ 85%阈值触发飞书通知
- ✅ 1小时冷却期（避免重复通知）

### 技术细节
**脚本位置**：`~/.openclaw/workspace/tools/context-monitor.sh`

**核心改进**：
```bash
# 旧版：数文件（不准确）
RECENT_MESSAGES=$(find "$HOME/.openclaw/workspace/memory" -name "*.md" -mmin -60 2>/dev/null | wc -l)

# 新版：调用API（准确）
sessions_json=$(openclaw sessions --active 120 --json 2>&1)
total_tokens=$(echo "$sessions_json" | jq '.sessions[0].totalTokens')
context_tokens=$(echo "$sessions_json" | jq '.sessions[0].contextTokens')
usage=$((total_tokens * 100 / context_tokens))
```

**配置参数**：
- 检查频率：每10分钟（Crontab）
- 阈值：85%
- 冷却期：1小时
- 通知渠道：飞书（user:ou_64e8948aedd09549e512218c96702830）

### 监控机制（完整版）
```
定时任务（每10分钟）
  ↓
调用OpenClaw API
  ↓
获取会话信息（totalTokens / contextTokens）
  ↓
计算使用率
  ↓
≥85%？
  ├── 是 → 检查冷却期 → 发送飞书通知
  └── 否 → 记录日志
```

### 剩余问题
1. **10分钟间隔仍可能错过爆点**
   - 改进方案：缩短到5分钟或3分钟
   - 长期方案：OpenClaw内置AI主动检测（每次回复前检查）

2. **无法真正阻止超限**
   - 只能提前提醒用户
   - 无法自动创建新会话（需要agentTurn机制，但需要OpenClaw支持）

### 后续优化方向
- [ ] 缩短检查间隔（10分钟 → 5分钟）
- [ ] 增加多会话监控（不只是第一个会话）
- [ ] 支持自定义阈值（不同模型不同阈值）
- [ ] 实现AI主动检测（每次回复前检查）
